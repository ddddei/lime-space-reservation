# 지시서 ⑩: 참가자/관리자 가이드 전면 개정 + 앱 내 도움말 화면

## 1. 배경과 전제

- `USER_GUIDE.md`(참가자)·`ADMIN_GUIDE.md`(관리자)는 초기 골격에 기능이 추가될 때마다 절을 덧붙여 와서 흐름이 매끄럽지 않다. **현재 기능 전체를 기준으로 처음부터 다시 쓴다.**
- 추가로 앱 안에서 바로 볼 수 있는 **도움말 화면**(참가자용/관리자용)을 만든다 — 문서의 요약본을 화면에 탑재한다.
- 개정된 USER_GUIDE.md는 이후 공유용 PDF의 원천이 된다 (PDF 제작은 이 지시서 범위 아님).
- 이 작업이 아닌 것: README 등 다른 운영 문서 개정, 기능 동작 변경, PDF 생성.

## 2. 작업 전 필독 파일 (읽지 않고 쓰지 마라)

문서를 쓰기 전에 **실제 화면 코드로 현재 동작을 확인**한다. 문서가 코드와 어긋나면 안 된다.

- `USER_GUIDE.md`, `ADMIN_GUIDE.md`, `FAQ.md`, `README.md` — 기존 문체·용어(예: "모임공간 신청하기", "참여자 확인")
- `src/components/UserReservationFlow.tsx`, `SpaceLanding.tsx`, `SpaceDetail.tsx`, `TimeBlockSelector.tsx`, `CalendarView.tsx`, `MyMeetings.tsx`, `ParticipantLogin.tsx` — 참가자 흐름
- `src/components/AdminPage.tsx`, `AdminUserChecklist.tsx`, `AdminAccountChecklist.tsx`, `SpaceAdminEditor.tsx`, `SpaceCreateForm.tsx`, `AdminBlockForm.tsx`, `AdminReservationTable.tsx` — 관리자 흐름
- `src/data/settings.ts` — 한도 값(30분 블록, 하루 4시간, 모임당 6회차, 레벨 한도, 6개월 달력, 기수 목록)

## 3. 산출물

### 3-1. USER_GUIDE.md 전면 개정 (참가자용)

구성 (이 순서로):
1. 로그인 (이름+전화번호, 승인 후 예약 가능)
2. 남은 시간 확인 (상단 배지, 레벨별 한도: Level 1 = 8시간, Level 2 = 24시간 — settings.ts 값으로 검산해 시간 단위로 표기)
3. 공간 둘러보기 (사진 여러 장 보기 포함)
4. 예약하기 (달력 오늘부터 6개월, 30분 단위 선택, 하루 최대 4시간, 하루 한 공간, 모임당 최대 6회차, 24시간 공간은 09~22시)
5. 내 신청 확인/수정/취소
6. 자주 묻는 질문 몇 가지 (기존 FAQ에서 참가자 관련 핵심만 요약, FAQ.md 링크)
- 운영자가 참가자에게 그대로 공유할 수 있는 눈높이로. 기술 용어 금지. 기존 문서의 존댓말 문체 유지.

### 3-2. ADMIN_GUIDE.md 전면 개정 (관리자용)

구성:
1. 관리자 로그인 / 계정 관리 (추가·비활성·재활성, 잠금 방지 규칙)
2. 참가자 관리 (추가·비활성·재활성, 승인 토글, Level, **기수(1기/2기) 관리**, **사용시간 초기화** — 기록을 지우지 않고 계산 기준일만 바꾼다는 설명 필수, "2기 시작 시점에 사용" 권고 포함)
3. 신청 관리 (전체 목록, 취소, 취소된 신청 보기)
4. 공간 관리 (새 공간 추가 — 사진 업로드·운영시간 프리셋/직접 설정, 기존 공간 수정 — 사진 교체·갤러리 여러 장·대표 지정·운영시간 수정, 사용자 숨김/비활성의 차이)
5. 차단 일정
6. 디스코드 알림 (신청/취소 시 자동 발송, 채널 변경 방법은 웹훅에서)
7. 문제가 생겼을 때 (TROUBLESHOOTING.md 링크)

### 3-3. 앱 내 도움말 화면

- **참가자 화면**: 상단(또는 Discord 문의 카드 근처)에 "이용 안내" 버튼 → 모달로 3-1의 요약(로그인/남은시간/예약 규칙/취소 방법, 스크롤 가능). 기존 모달 스타일 관례 재사용.
- **관리자 화면**: 헤더 부근 "도움말" 버튼 → 모달로 3-2의 요약(섹션별 한 문단씩).
- 도움말 내용은 하드코딩된 상수 파일(`src/data/helpContent.ts` 등)로 분리해 문서 개정 시 같이 고치기 쉽게 한다. 새 의존성 금지, 기존 스타일 토큰(색상 클래스) 관례 준수.
- 한도 수치는 가능한 한 `settings.ts` 상수에서 계산해 표기 (문서와 화면이 어긋나지 않게).

### 3-4. FAQ.md 소정비

- 전면 개정은 아니고, 현행과 어긋난 항목(1시간 단위, 달력 범위 등)만 30분 단위·6개월 기준으로 수정.

## 4. 금지 사항

- git 커밋/스테이징/브랜치 변경 금지. 새 npm 의존성 금지.
- `supabase/` 수정 금지. 기능 동작 변경 금지 (도움말 UI 추가만).
- README.md·QUICK_START 등 다른 문서 대량 수정 금지 (링크 정합만 필요 시 최소 수정).
- `dist/`, `node_modules/`, `.env.local` 접근 금지.

## 5. 완료 기준

1. `npx tsc --noEmit` 통과. 2. `npm run build` 통과. 3. `npx eslint src` 통과 (워닝 증가 없음).
4. 문서 검산: USER_GUIDE의 모든 수치(30분, 4시간, 6회차, 레벨 한도, 6개월, 09~22시)가 `settings.ts`·코드와 일치하는지 항목별 대조표를 보고에 포함.
5. `npm run dev` 수동 확인(종료 필수): 참가자 "이용 안내" 모달 열기/닫기, 관리자 "도움말" 모달 열기/닫기, 모바일 폭(375px)에서 모달 스크롤 정상.
6. 기존 문서 대비 사라진 정보가 없는지 확인 (기존 절 목록 대비 커버리지 표).

**보고에 포함할 것**: 변경 파일 목록, 검증 명령별 결과, 수치 대조표, 커버리지 표, 수동 확인 결과, 판단 지점, 보류 항목, `git status --short`.
